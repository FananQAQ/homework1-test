# 技术总结

## 1. 完成内容概览

- 使用 **llama.cpp** 源码，通过仓库根目录 **CMakeLists.txt** 集成构建，产出 **`local_llm.exe`**（与 `llama-completion` 等价，满足命名要求）。
- **性能**：`tests/perf_smoke_test.py` 实现 ≥10 组不同长度输入，统计 **E2E**、**TPOT**，并给出 **TTFT 的工程估计**；可选 **matplotlib** 生成柱状图 PNG。
- **准确度**：`tests/gen_appointment_dataset.py` 构造 100 正例 + 100 反例；`tests/accuracy_eval.py` 调用本地模型并计算 **准确率、精确率、召回率、F1**。

## 2. 遇到的问题与解决

| 问题 | 解决方案 |
|------|----------|
| `need.md` 要求 `local_llm.exe`，而上游目标名为 `llama-completion` | 根目录 CMake 增加依赖目标 `local_llm_exe`，在构建后 `copy_if_different` 到同目录 `local_llm(.exe)`。 |
| 将 llama.cpp 作为子工程时默认不编 `common/tools` | 在父工程 `CACHE FORCE` 打开 `LLAMA_BUILD_COMMON`、`LLAMA_BUILD_TOOLS`，关闭 `TESTS/EXAMPLES/SERVER` 以控编译量。 |
| 父目录无法对子目录目标使用 `POST_BUILD` 式 `add_custom_command(TARGET …)` | 改用顶层 `add_custom_target(... DEPENDS llama-completion)` 执行复制命令。 |
| CLI 无流式首 token 时间 | 文档与脚本中明确 TTFT 为 **估计值**；若课程要求严格 TTFT，可改用 `llama-server` + 流式客户端计时。 |
| Windows 批处理在 `if ()` 块内 `goto` 导致解析错乱 | 合并为单一 `run_all.bat`，避免块内 `goto`。 |

## 3. 收获

- 熟悉 **CMake** 多工程嵌套与 **MSVC** 多配置输出路径（`bin/Release`）。
- 理解 **prefill（prompt eval）** 与 **decode（eval per run）** 在 `llama.cpp` 性能日志中的含义，便于对接 TTFT/TPOT 指标。
- 掌握用 **Python + 子进程** 对黑盒 CLI 做批量基准与简单 **NLP 二分类**评测的流程。

## 4. 后续可改进方向

- 替换为中文能力更强的 GGUF，并固定 `temperature`、`seed` 以提高可复现性。
- 使用 **grammar / JSON** 约束输出为 `{"ok":true}` 形式，降低解析失败。
- 将 200 条评测改为 **批处理或 server 长连接**，缩短总耗时。
